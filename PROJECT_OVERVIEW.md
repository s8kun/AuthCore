# Project Overview

## Product Summary

Auth-as-a-Service is a multi-project authentication platform built with Laravel, Filament, Sanctum, and Livewire.

Its job is to let a platform owner create client projects, assign each project its own auth configuration and mail setup, and expose one project-scoped API for end-user authentication workflows.

The current product surface is intentionally focused:

- registration
- login
- current-user lookup
- logout
- refresh-token rotation
- OTP send, verify, and resend
- forgot-password request and password reset completion
- ghost-account invitation and claim
- project-level custom user field definitions
- project-level mail templates and delivery settings
- project-level observability through request and auth logs

## Current Runtime Surfaces

- `/` redirects to `/admin`
- `/admin` is the Filament control plane
- `/api/v1/auth` is the public auth API
- `/docs/api` and `/docs/api.json` are generated by Scramble
- `/admin/docs/project-docs` is the global docs landing page inside the panel
- each project also has panel pages for integration details and API reference

## Tenant And Identity Model

### Platform Users

Platform users live in the default `users` table.

They:

- sign in to the Filament panel
- own projects
- manage auth settings, mail settings, templates, project users, and logs

### Projects

A `Project` is the tenant boundary for the public API.

Each project has:

- a UUID primary key
- `name`, `slug`, `api_key`, `api_secret`, `status`, and `rate_limit`
- one auth settings record
- one mail settings record
- many email templates
- many project users
- many custom field definitions and values
- many OTPs, password resets, refresh tokens, and logs

Projects automatically provision default auth settings, mail settings, and email templates when they are created.

### Project Users

Project users live in `project_users` and are the end users of a client project.

Important built-in fields include:

- `email`
- `password`
- `email_verified_at`
- `last_login_at`
- `is_active`
- `is_ghost`
- `claimed_at`
- `invited_at`
- `ghost_source`
- `must_set_password`
- `must_verify_email`

The uniqueness boundary is `project + email`, not global email across the whole platform. The same email can exist in multiple projects.

Profile-style fields such as `first_name`, `last_name`, and `phone` are no longer stored directly on `project_users`. They now belong in the custom project user schema and are submitted under `custom_fields`.

## Public API Shape

The public API is mounted under:

```text
/api/v1/auth
```

Current endpoints:

- `POST /register`
- `POST /login`
- `POST /refresh`
- `POST /forgot-password`
- `POST /reset-password`
- `POST /send-otp`
- `POST /verify-otp`
- `POST /resend-otp`
- `POST /ghost-accounts`
- `POST /ghost-accounts/claim`
- `GET /me`
- `POST /logout`

Every request must include:

```http
X-Project-Key: {project_api_key}
```

Protected routes also require:

```http
Authorization: Bearer {access_token}
```

## Request Pipeline

Every public auth route passes through:

1. `ResolveProjectFromApiKey`
2. `LogProjectApiRequest`
3. `throttle:project-auth`

Authenticated routes then add:

4. `auth:sanctum`
5. `EnsureProjectAccessMatchesToken`

### What This Enforces

`ResolveProjectFromApiKey`:

- requires the `X-Project-Key` header
- resolves the project from `projects.api_key`
- rejects inactive projects

`EnsureProjectAccessMatchesToken`:

- rejects missing or invalid bearer tokens
- rejects tokens issued by another project
- rejects inactive project users
- rejects users who are still pending email verification

The top-level route rate limiter uses `projects.rate_limit` and keys limits by project, IP address, and endpoint. The default per-project rate limit is `60` requests per minute.

## Authentication Model

The current public API does not use JWT.

It uses:

- short-lived Sanctum personal access tokens
- long-lived custom refresh tokens stored in `refresh_tokens`

Refresh tokens are:

- hashed at rest
- scoped to project and project user
- rotated on successful refresh
- revoked on logout
- revoked on password reset
- fully revoked if refresh token reuse is detected

## Registration Flow

Registration is handled by `ProjectAuthService` and is always project-scoped.

Current behavior:

- a new email creates a new `project_users` record
- an existing pending user is retried on the same record
- an existing ghost account blocks normal registration
- an existing verified user returns a `422` validation error
- `custom_fields` are validated against the active project schema
- unknown custom field keys are rejected
- field defaults are applied when configured
- unique custom fields are enforced within the current project

When email verification is enabled:

- registration returns `202 Accepted`
- no tokens are returned
- an email-verification OTP is queued
- old access and refresh tokens for that pending user are revoked

When email verification is disabled:

- registration returns `201 Created`
- the user is active immediately
- a token pair is issued
- a welcome email is queued

## Login, Logout, And Current User

Login succeeds only when the project user:

- exists in the current project
- is active
- is not a ghost account
- has a password
- passes password verification
- is not pending email verification

On successful login:

- `last_login_at` is updated
- a new access token is issued
- a new refresh token is created
- a login success event is logged

Logout revokes:

- the current access token
- all active refresh tokens for that project user

`GET /me` returns the authenticated project user resource with API-visible custom fields only.

## OTP System

The OTP system is project-scoped and purpose-scoped.

Current purposes:

- `register_verify`
- `login_verify`
- `forgot_password`
- `ghost_account_claim`
- `email_verification`

Each OTP record stores:

- hashed code
- email
- purpose
- optional `project_user_id`
- expiry timestamp
- attempt counters
- resend counters
- last sent timestamp
- consumed timestamp
- metadata

Delivery behavior:

- OTP emails are queued
- email template selection depends on purpose
- resend cooldown is enforced
- daily per-email and per-purpose limits are enforced
- invalid or expired codes return `422`

## Password Reset Flow

Password resets are project-scoped.

Current behavior:

- forgot-password is optional per project
- requests are rate-limited per project and per email
- the response is intentionally generic even when the user is missing
- active, non-ghost users receive a reset email with a tokenized link
- successful reset updates the password, clears `must_set_password`, revokes active tokens, and queues a reset-success email

## Ghost Accounts

Ghost accounts are invitation-style project users managed by `GhostAccountService`.

Creation behavior:

- only available when `ghost_accounts_enabled` is true
- creates or updates a ghost `project_user`
- supports `must_set_password`, `must_verify_email`, `ghost_source`, and `custom_fields`
- can optionally send a claim invite immediately

Claim behavior:

- requires the invited email and a valid `ghost_account_claim` OTP
- can optionally require a password
- converts the user from ghost to normal account
- marks the account as verified
- issues a fresh access token and refresh token pair

## Custom Project User Schema

Projects can define custom user fields through `project_user_fields`.

Supported field types:

- `string`
- `text`
- `integer`
- `decimal`
- `boolean`
- `date`
- `datetime`
- `enum`
- `email`
- `url`
- `phone`
- `uuid`
- `json`

Field definitions support:

- stable `key` and human `label`
- optional description and placeholder
- optional default values
- enum options
- type-specific validation rules
- required, nullable, and unique behavior
- search, filter, and sort toggles
- API visibility, admin-form visibility, and table visibility
- active / archived lifecycle
- explicit ordering

Custom values are stored in typed columns on `project_user_field_values`. Only fields marked `show_in_api` appear in API responses.

Reserved keys include all built-in auth columns plus `custom_fields`, so schema-defined fields cannot collide with the core account model.

## Mail And Templates

Each project has one `project_mail_settings` record.

Current delivery modes:

- `platform`
- `custom_smtp`

Mail settings cover:

- sender identity
- reply-to and support addresses
- SMTP host, port, username, password, encryption, and timeout
- verification status and last-tested timestamp

The panel supports sending a test email and clearing the stored SMTP password.

Each project also has branded templates for:

- `otp`
- `forgot_password`
- `reset_password_success`
- `welcome`
- `email_verification`
- `ghost_account_invite`

Available placeholders include:

- `{{ project_name }}`
- `{{ user_email }}`
- `{{ otp_code }}`
- `{{ reset_link }}`
- `{{ support_email }}`
- `{{ expires_in }}`
- `{{ app_name }}`

## Observability

Two logging layers are built in:

### API Request Logs

Every public auth request is logged with:

- project
- endpoint
- route name
- method
- email when available
- IP address
- user agent
- status code
- success flag
- metadata

The logging middleware runs before the response is finalized, so throttled and failed requests are still visible.

### Auth Event Logs

Domain-level auth events are also logged, including:

- registration success and failure
- login success and failure
- logout success
- OTP sent, resent, verified, and failed
- password reset requested and completed
- verification sent and completed
- ghost account created and claimed
- refresh rotation outcomes

## Current Implementation Notes

- `X-Project-Key` is the only project credential used by current public routes
- `api_secret` is generated and stored but not used in the public auth middleware yet
- `magic_link_enabled` exists in auth settings, but no magic-link routes are currently exposed
- generated Scramble docs exist alongside the curated Markdown docs in this repository
